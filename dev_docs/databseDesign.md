# Entity-Relationship Diagram (ERD) & Database Design Document

## 1. Architectural Foundations
* **Multi-Tenancy & Data Isolation:** The root of the system maps ownership to a standard User model through a dedicated many-to-many relationship table (`shop_owners`). Employees (Cashiers, Managers) are directly bound to individual physical operations using a nullable `shop_id` foreign key.
* **Ledger-First Architecture:** Transactions are written as immutable records. Changes to physical or digital currency levels are recorded as discrete transaction lines to maintain end-of-day compliance balances.
* **Client-Side UUID Primary Keys:** Tables use UUID primary keys generated by the React Native edge client to support offline micro-batch synchronization and prevent primary key collision or duplication.

## 3. Table Specifications & Dictionary

### 2.1 Table: `users`
Represents platform authenticators, administrators, business owners, and staff members.
* `shop_id` is nullable, which resolves registration dependencies for multi-shop owners before an operation is provisioned.

| Field | Type | Attributes | Description |
| :--- | :--- | :--- | :--- |
| `id` | `UUID` | `PRIMARY KEY` | Unique identifier generated on the client. |
| `shop_id` | `UUID` | `NULLABLE`, `INDEX` | Active workplace roster location for shifts. |
| `name` | `VARCHAR(255)` | `NOT NULL` | Full legal or display name. |
| `email` | `VARCHAR(255)` | `UNIQUE`, `NOT NULL` | System login indicator. |
| `password` | `VARCHAR(255)` | `NOT NULL` | Hashed authentication password. |
| `role` | `VARCHAR(50)` | `NOT NULL` | Role matching the `UserRole` Enum. |
| `pin` | `VARCHAR(255)` | `NULLABLE` | Hashed numeric lock-screen pin for swift terminal switches. |
| `timestamps` | `TIMESTAMP` | `NULLABLE` | Standard `created_at` and `updated_at`. |

### 2.2 Table: `shops`
The main tenant grouping that holds catalog data and transactional configurations.

| Field | Type | Attributes | Description |
| :--- | :--- | :--- | :--- |
| `id` | `UUID` | `PRIMARY KEY` | Main identity token. |
| `name` | `VARCHAR(255)` | `NOT NULL` | Operational storefront name. |
| `shop_type` | `VARCHAR(50)` | `NOT NULL` | Type matching the `ShopType` Enum (e.g., Bar, Retail). |
| `latitude` | `DECIMAL(10,8)` | `NOT NULL` | Spatial location parameter. |
| `longitude` | `DECIMAL(11,8)` | `NOT NULL` | Spatial location parameter. |
| `allowed_radius` | `INT` | `NOT NULL` | Geofence restriction radius in meters. |
| `timestamps` | `TIMESTAMP` | `NULLABLE` | Standard database timestamps. |

### 2.3 Table: `shop_owners`
The many-to-many bridge table that allows multiple partners to share ownership of multiple storefront portfolios.

| Field | Type | Attributes | Description |
| :--- | :--- | :--- | :--- |
| `id` | `BIGINT` | `PRIMARY KEY`, `AUTO_INC` | Internal sequence index. |
| `shop_id` | `UUID` | `NOT NULL`, `FOREIGN KEY` | References `shops.id` (`ON DELETE CASCADE`). |
| `user_id` | `UUID` | `NOT NULL`, `FOREIGN KEY` | References `users.id` (`ON DELETE CASCADE`). |
| `timestamps` | `TIMESTAMP` | `NULLABLE` | Association tracking timestamps. |

* **Constraints:** `UNIQUE(shop_id, user_id)` to prevent duplicate ownership assignments.

| Field | Type | Attributes | Description |
| :--- | :--- | :--- | :--- |

| `shop_id` | `UUID` | `NOT NULL`, `FOREIGN KEY` | References `shops.id`. |
| `user_id` | `UUID` | `NOT NULL`, `FOREIGN KEY` | References `users.id`. |

| `blind_swipe_reported`  | `DECIMAL(10,2)` | `NULLABLE` | Swipe card balances declared at closing. |

## 4. Reference Notes for Future Maintenance Developers

1. **Why `DECIMAL(10,3)` for Quantities?** This precision supports fractional kitchen metrics (e.g., writing off `0.450 kg` of beef fillet) and fractional bar configurations (e.g., tracking partial open pour lines after an evening shift count).
2. **Pessimistic Write Locking Strategy:**
   When completing transactions or updating tab entries, developers must use a row-level lock (`SELECT ... FOR UPDATE` via Laravel's `lockForUpdate()`) inside database transactions. This safeguards system balances against duplicate processing or race conditions caused by network retries from the mobile app.
3. **Data Cascades:**
   Deletions on core tenant catalogs (`shops`, `products`) trigger cascading deletes across dependent operational assets (`shop_owners`, `stocks`, `bottles`, `units`, `shot_sizes`). However, transactional records like `orders` and `shifts` do not use cascading deletes to preserve data integrity for auditing and historical financial analysis.
